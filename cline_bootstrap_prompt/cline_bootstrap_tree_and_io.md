你現在要協助我整理一個程式碼專案，目標是讓「未來的我」回來時，可以很快理解：

1. 專案的目錄與檔案結構（重點是入口與核心模組，而不是全部檔名）。
2. 這個專案的 Input → Execution → Output 流程，以及輸出檔案大致會放在哪裡。

請依照下列步驟行動：

### 步驟 1：建立「目錄與檔案結構」摘要

1. 掃描專案目錄，但**不要**逐檔解說。
2. 產出一段 Markdown，對應 `PROJECT_NAVIGATOR.md` 的：

   - `1.1 高層目錄樹`：列出 2～3 層的重要目錄，忽略 `node_modules`, `.git`, `__pycache__` 等雜項。
   - `1.2 重要檔案索引`：列出：
     - 執行入口（例如：`main.py`, `app.tsx`, `cli.py`）
     - 商業邏輯或核心流程檔案
     - 重要設定檔（LLM、工具、路徑等）

### 步驟 2：建立「Input → Execution → Output」摘要（使用 Flow Descriptions）

1. 根據程式碼與現有文件，整理 3～5 個具有代表性的「執行路徑」（例如：CLI、HTTP API、批次腳本等不同模式），但在提示中不需要展開具體例子。
2. 對每一條執行路徑，請用文字描述一個 **Flow**，包含三個區塊：
   - **Input：** 說明輸入的來源與大致形式。
   - **Execution：** 依序描述主要經過的檔案、模組或流程（點到為止，不要整段貼程式碼）。
   - **Output：** 說明輸出的大致型態（文字、JSON、檔案等），以及主要會寫入哪些目錄（例如 `logs/`, `output/`, `tmp/`）。

3. 用 Markdown 形式輸出這些 Flow 描述，讓它們可以對應到 `PROJECT_NAVIGATOR.md` 中的：
   - Input → Execution → Output 流程描述區塊
   - 輸出檔案結構說明（以文字列出常用輸出目錄與用途）

### 回答風格要求

- 回答時**不要嘗試列出完整專案樹**，只要足夠讓人知道：
  - 哪些目錄是「骨架」
  - 哪些檔案是「入口點」
  - 哪些流程會產生實際輸出
- 描述 Flow 時，用簡潔的文字分段說明 `Input / Execution / Output` 即可，不需要使用表格。
- 如果資訊不夠完整（例如沒有現成輸出目錄），請做合理推測並清楚標註「推測」。
